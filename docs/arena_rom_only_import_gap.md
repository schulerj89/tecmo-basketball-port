# Arena Intro ROM-Only Import Gap

This note explains why the arena and post-arena intro frames cannot currently be
finished from a `.nes`-only asset pack, what external capture state the native
renderers still consume, and how to replace that capture dependency with a
ROM-only extractor.

## Current State

`--build-assetpack` is intentionally raw-ROM only. `tecmo_asset_pack_build_from_ines`
validates iNES and writes `system/manifest`, `prg/bankNN`, `prg/fixed`,
`chr/all`, `chr/bankNN`, and `system/source-map` from the ROM bytes
(`src/tecmo_asset_pack.c:922`, `src/tecmo_asset_pack.c:980`,
`src/tecmo_asset_pack.c:1016`, `src/tecmo_asset_pack.c:1059`,
`src/tecmo_asset_pack.c:1113`). The import workflow explicitly excludes
`intro/arena/capture.ndjson`, `intro/post-arena/capture.ndjson`, and
`intro/captures/source-map` until those logical entries are generated by a
ROM-only extractor (`docs/assetpack_import_workflow.md:58`).

Runtime still has capture loaders for those old logical entries and loose
debug files. Arena capture lookup accepts `intro/arena/capture.ndjson`,
`intro/arena/emu_intro_memory_watch.ndjson`,
`intro/arena/emu_intro_arena_irq_watch.ndjson`, and similarly named loose files
(`src/tecmo_intro_arena.c:57`, `src/tecmo_intro_arena.c:1092`,
`src/tecmo_intro_arena.c:1128`). Post-arena lookup accepts
`intro/post-arena/capture.ndjson` and memory-watch logs
(`src/tecmo_intro_post_arena.c:33`, `src/tecmo_intro_post_arena.c:815`,
`src/tecmo_intro_post_arena.c:851`).

The renderers therefore have CHR graphics from `chr/all`, but not the runtime
PPU, OAM, mapper, scroll, or palette state needed to decide which CHR tiles are
visible at each intro frame. When those captures are absent, the arena, READY,
and WARRIORS render modes intentionally draw missing-capture diagnostics
(`src/tecmo_game.c:1881`, `src/tecmo_game.c:1945`, `src/tecmo_game.c:1989`).

## External Capture Inputs

### Arena Screen

`load_arena_capture_text` consumes NDJSON rows in a narrow frame window
(`src/tecmo_intro_arena.c:912`):

| Input kind | Current frame window | Stored state | Renderer consumer |
| --- | --- | --- | --- |
| `ppu_nametable_write_batch` / `nametable_writes` | 428-430 | Final tile values for PPU `$2000-$27BF`, split into two nametable pages | `tecmo_intro_arena_draw_composite` draws the staged pages (`src/tecmo_intro_arena.c:944`, `src/tecmo_intro_arena.c:1385`) |
| `ppu_attribute_write_batch` / `attribute_writes` | 427-430 | Attribute bytes for `$23C0-$23FF` and `$27C0-$27FF` | `arena_draw_tile_entry` selects background palette quadrant (`src/tecmo_intro_arena.c:950`, `src/tecmo_intro_arena.c:1296`) |
| `ppu_palette_write_batch` / `palette_writes` | 425-476 | Timed background and sprite palette stages, offset from first capture frame | `tecmo_intro_arena_palette_for_frame` and sprite palette selection (`src/tecmo_intro_arena.c:956`, `src/tecmo_intro_arena.c:1231`) |
| `oam_frame_diff` / `visible_oam` | 461-811 | Visible 8x16 sprite stages with Y, tile, attributes, X | `tecmo_intro_arena_draw_sprites`, including basket lag handling (`src/tecmo_intro_arena.c:961`, `src/tecmo_intro_arena.c:1569`) |
| `ppu_arena_rows_snapshot` / `rows`, `ranges`, `oam` | capture watcher offsets | Backfill nametable rows, attributes, palettes, and basket OAM from PPU memory snapshots | Same tile, palette, and sprite consumers (`src/tecmo_intro_arena.c:966`) |

Arena capture also needs MMC3 CHR context that is currently partly hard-coded
and partly overridable with environment variables: upper background R0/R1,
lower background R0/R1, split row, page CHR banks, and sprite CHR bank
(`src/tecmo_intro_arena.c:26`, `src/tecmo_intro_arena.c:1141`). The actual
tile-to-CHR address mapping uses those R0/R1 values in `arena_mmc3_bg_tile_offset`
(`src/tecmo_intro_arena.c:125`). The narrow FCEUX watcher was created exactly to
observe mapper writes, IRQ writes, scroll pairs, row snapshots, and compact OAM
facts around this screen (`tools/emu_intro_arena_irq_watch.lua:4`).

Frame timing is also captured-derived. The native render frame is mapped to the
observed arena capture timeline by `tecmo_intro_arena_display_frame`
(`src/tecmo_intro_arena.c:68`), and the renderer scrolls the composite through
`tecmo_intro_arena_transition_state` while using the captured background and
sprite stages (`src/tecmo_game.c:1839`).

### Post-Arena READY/WARRIORS Screens

`load_post_capture_text` consumes a longer frame interval from READY through
WARRIORS (`src/tecmo_intro_post_arena.c:660`,
`include/tecmo_intro_post_arena.h:13`):

| Input kind | Stored state | Renderer consumer |
| --- | --- | --- |
| `mapper_write_batch` / `mapper_writes` | Current MMC3 register state, copied into later tile events | `bg_tile_offset` and `sprite_tile_offset` resolve CHR offsets through the per-event/stage mapper (`src/tecmo_intro_post_arena.c:698`, `src/tecmo_intro_post_arena.c:994`) |
| `ppu_nametable_write_batch` / `nametable_writes` | Timed tile events with PPU address, tile value, and mapper snapshot | `build_scene_cells` replays events from scene start to target frame (`src/tecmo_intro_post_arena.c:701`, `src/tecmo_intro_post_arena.c:1096`) |
| `ppu_attribute_write_batch` / `attribute_writes` | Timed attribute-table writes | `draw_scene_tiles` picks palette quadrants (`src/tecmo_intro_post_arena.c:704`, `src/tecmo_intro_post_arena.c:1139`) |
| `ppu_palette_write_batch` / `palette_writes` | Timed background and sprite palette stages | `palette_for_frame` selects palette for READY/WARRIORS (`src/tecmo_intro_post_arena.c:707`, `src/tecmo_intro_post_arena.c:1047`) |
| `oam_frame_diff` / `visible_oam` | Timed visible sprite stages plus mapper state | WARRIORS draws the selected sprite stage (`src/tecmo_intro_post_arena.c:710`, `src/tecmo_intro_post_arena.c:1186`) |
| `ppu_scroll_write_batch` / `ppu_scroll_writes` | Timed horizontal scroll values | WARRIORS applies `scroll_x_for_frame` to page placement (`src/tecmo_intro_post_arena.c:717`, `src/tecmo_intro_post_arena.c:1080`) |

READY maps native frame `N` to capture frame `1034 + N`, clamped to 1091.
WARRIORS maps native frame `N` to capture frame `1092 + N`, clamped to 1305
(`src/tecmo_intro_post_arena.c:899`, `src/tecmo_intro_post_arena.c:908`). These
screens are not just static nametable dumps: the renderer reconstructs each
target frame by replaying timed writes and mapper snapshots (`src/tecmo_intro_post_arena.c:1242`).

## What Is Recoverable Statically From PRG/CHR

The raw `.nes` asset pack already contains:

- CHR pattern data for all background and sprite tiles via `chr/all` and
  `chr/bankNN`.
- PRG bytes for all switchable and fixed banks via `prg/bankNN` and `prg/fixed`.
- The iNES mapper number, PRG count, CHR count, and trainer size in
  `system/manifest`.

From those bytes, a static decoder should be able to recover:

- CHR banks and tiles referenced by known literal/script tables, once the table
  format is mapped.
- Pointer-table streams like the existing `A7DB` and `A90F` intro sprite stream
  decoders in `tools/Find-IntroRabbitLookup.ps1`,
  `tools/Find-IntroTecmoLogoLookup.ps1`, and
  `tools/Find-IntroCompositeTrace.ps1`.
- Fixed helper vectors and high-level intro handoff evidence like the boot and
  procedure mapping scripts already report
  (`tools/Find-BootFirstScreenTrace.ps1`, `tools/Find-IntroProcedureMapping.ps1`).
- Literal default constants, command tables, CHR bank numbers, palette table
  bytes, pointer targets, and stream record shapes.

Static decode alone is not enough for the arena/post-arena PNGs because the
needed state is the result of executing those tables and routines over hundreds
of frames.

## What Requires Execution Or Simulation

These values are dynamic and should be generated by replaying the intro logic:

- Final arena nametable pages and attribute bytes after all queued PPU writes,
  including row snapshots that came from live PPU memory rather than a single
  obvious table.
- Timed palette stages, including carry-forward behavior across frames.
- Visible OAM stages after sprite staging, cleanup, DMA timing, and frame-to-frame
  differences.
- MMC3 register values at the exact draw/write time, including arena upper/lower
  background banks, post-arena per-tile mapper snapshots, sprite mapper context,
  and any IRQ/scanline split setup.
- PPUSCROLL/PPUCTRL-derived scroll values and nametable selection.
- Frame offsets between native render-test counters and original intro capture
  frames.
- Any state seeded through zero page/RAM variables such as the existing watcher
  state signatures and the Bank04 intro sequence state.

The current FCEUX scripts prove this dependency: `emu_intro_memory_watch.lua`
logs writes to `$2000-$2007`, `$8000/$8001`, scroll pairs, palette writes,
nametable/attribute batches, and visible OAM every frame; the arena-specific
watcher waits for an MMC3 R0/R1 signature and then captures mapper, IRQ, scroll,
OAM, and PPU row snapshots.

## Recommended ROM-Only Extractor Architecture

Use a three-layer extractor and keep its output as logical asset-pack entries.

1. Static table decode

   Read PRG/CHR entries from the in-memory iNES/asset-pack image and expose
   helpers for banked CPU reads, fixed-bank reads, pointer-table reads, CHR bank
   slices, and safe source-map annotations. Port the useful parts of the
   existing `Find-*` scripts from PowerShell into C or a build-time tool module.
   This layer should emit table metadata and candidate records, but it should
   not claim final arena frames.

2. Minimal NES CPU/PPU/MMC3 trace replay

   Add a deterministic replay mode that boots the ROM, runs the intro until the
   required frame ranges, and records the same neutral events the C renderers
   already parse: nametable writes, attribute writes, palette writes, mapper
   writes, scroll writes, visible OAM stages, and selected PPU snapshots. This
   does not need a full user-facing emulator, but it does need enough 6502, PPU
   register, OAM DMA, CPU RAM, PRG bank switching, CHR bank switching, MMC3 IRQ,
   NMI, and frame timing behavior to reproduce the intro scripts.

3. Logical entry writer

   Convert replay output into `intro/arena/capture.ndjson`,
   `intro/post-arena/capture.ndjson`, and `intro/captures/source-map` inside the
   asset pack. The renderers can keep their current parsers at first. Once the
   replay data is stable, replace loose-file fallbacks and migrate the text
   NDJSON into a compact binary schema if desired.

Avoid a hand-ported C version of only the arena routines as the first solution.
It would require porting Bank04/fixed helper behavior, scheduler state, PPU
queues, mapper side effects, OAM staging, scroll writes, and IRQ split behavior
before it could be trusted. That path is useful later for native runtime
fidelity, but it is the highest-risk way to close the importer gap.

## Recommended First Step

Implement the minimal replay tracer first, but make the first milestone narrow:

- Input: the same `.nes` passed to `--build-assetpack`.
- Output: an ignored local trace with the same event kinds accepted by
  `load_arena_capture_text` and `load_post_capture_text`.
- Stop condition: produce enough events for `intro-arena-frame320`,
  `intro-ready-frame35`, and `intro-warriors-frame74` to render without loose
  FCEUX logs.
- Validation: compare event counts and PNG behavior against existing local
  FCEUX-derived captures, then write the logical entries into the pack and
  enable the skipped ROM-only render smoke in `tools/Run-IntroSequenceTests.ps1`.

This approach reuses the current renderer contracts, keeps ROM payload bytes out
of docs, and gives a direct path from raw PRG/CHR to the missing arena intro
state.
